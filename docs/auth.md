# Authentication and Sessions

TurnBasedEQ implements account-based authentication with salted password hashing and session tokens brokered by WorldLogin. Zone servers validate session token hashes on gameplay connections but do not perform password checks directly.

See also: [networking.md](networking.md) (login packets), [data-models.md](data-models.md) (DB schema).

| Module | Path | Responsibility |

## Password hashing

Implementation: `tbeq::auth::hashPassword()` / `verifyPassword()`.

| Property | Value |
|----------|-------|
| Algorithm | Custom iterative FNV-1a over `salt + password` |
| Iterations | 12,000 |
| Salt | 16 random bytes, hex-encoded |
| Hash output | 64-bit hex string |

```cpp
// shared/include/tbeq/auth/PasswordHash.hpp
struct PasswordHashResult
{
    std::string hash;
    std::string salt;
};
```

Password hashes and salts are stored in the `accounts` table (`password_hash`, `password_salt`).

> **Note:** This is a development-grade hash suitable for Phase 0. Production deployments should migrate to a standard password KDF (e.g. Argon2, bcrypt).

## Account validation

`CharacterValidation.hpp` provides:

| Function | Rules (summary) |
|----------|-----------------|
| `validateUsername` | Length and character set constraints |
| `validatePassword` | Minimum length requirements |
| `validateCharacterName` | Name format rules |
| `validateCharacterCreate` | Name + valid race/class from JSON catalogs |

Race and class JSON arrays are loaded at WorldLogin startup from `data/races.json` and `data/classes.json`.

## Session model

```mermaid
sequenceDiagram
    participant C as Client
    participant WL as WorldLogin
    participant DB as SQLite

    C->>WL: LoginRequest (username, password)
    WL->>DB: findAccountByUsername
    WL->>WL: verifyPassword
    WL->>WL: issueSession(accountId)
    WL->>DB: createSession(tokenHash, accountId, expiresAt)
    WL->>C: LoginResponse (success, raw token)

    Note over C,WL: Client stores token; zone handoff uses token hash
```

### Token lifecycle

1. WorldLogin generates a random 64-bit token value.
2. Token is hashed with FNV-1a over its decimal string representation (`sessionTokenHash()`).
3. **Hash** is stored in SQLite; raw token is sent to the client once at login.
4. Session expiry: 24 hours from issue (`expiresAt = now + 86400`).
5. Expired sessions are purged on new session issue.

The client sends `sessionTokenHash` (not the raw token) on zone `SessionResume` and in packet headers for authenticated gameplay packets.

## Login packet flow

| Step | Client packet | Server response |
|------|---------------|-----------------|
| 1 | `LoginRequest` | `LoginResponse` |
| 2 | `CreateAccountRequest` (optional) | `CreateAccountResponse` |
| 3 | `CharacterListRequest` | `CharacterListResponse` |
| 4 | `CreateCharacterRequest` (optional) | `CreateCharacterResponse` |
| 5 | `SelectCharacterRequest` | `ZoneConnectInfo` |

`SelectCharacterRequest` triggers server-server handoff:

- WorldLogin → destination zone: `PlayerEnterPrepare`
- Zone loads character via `LoadCharacterRequest` / `LoadCharacterResponse`
- Zone → WorldLogin: `PlayerEnterReady`
- WorldLogin → client: `ZoneConnectInfo` (host, port, character id, token hash)

## Character creation

New characters are created with:

- UUID `id` generated by `Database::createCharacter()`
- Starting zone from worldgen (typically `starter_city`)
- Default position and `CharacterState::createDefault(classId, level)`

Initial runtime state is serialized to `characters.state_json`.

---

## Session validation on zone servers

When a client connects to a zone server:

1. Client sends `SessionResume` with `characterId` and `sessionTokenHash`.
2. Zone server forwards validation to WorldLogin or trusts the handoff token from `PlayerEnterPrepare`.
3. On success, zone sends `SessionResumeResponse`, `ZoneSnapshot`, and initial `EntitySnapshot`.

Gameplay packets include `sessionTokenHash` in the 24-byte packet header for ongoing validation.

## Security boundaries

| Concern | Current behavior |
|---------|------------------|
| Transport encryption | None (plain TCP; local dev) |
| Token replay | Token hash in header; sessions expire |
| Zone authority | Zone servers do not accept login passwords |
| Dev commands | Gated by `--dev-mode` on servers |
